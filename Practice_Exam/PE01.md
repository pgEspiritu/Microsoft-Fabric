# Question 1

### Two tables in the data model are not connected in a physical relationship. You need to establish a virtual relationship between the tables.

- CROSSFILTER()
- PATH()
- ✅ **TREATAS()**
- USERELATIONSHIP()

## ✅ Correct Answer

**TREATAS()**

### Explanation

`TREATAS()` creates a **virtual relationship** by applying the values from one table as filters on another table.

Example:

```DAX
CALCULATE(
    [Sales],
    TREATAS(VALUES(TableA[CustomerID]), TableB[CustomerID])
)
```

Why not the others?

- **CROSSFILTER()** → Changes filter direction on an **existing** relationship.
- **USERELATIONSHIP()** → Activates an **inactive** relationship.
- **PATH()** → Works with parent-child hierarchies.

# Question 2

### Apply logic-based DAX formatting to only the Sales measure while minimizing administration.

- Calculation group measure formatting
- Conditional formatting
- ✅ **Dynamic measure formatting**
- Fields parameter

## ✅ Correct Answer

**Dynamic measure formatting**

### Explanation

**Dynamic measure formatting** changes the display format of **one measure** using DAX without affecting other measures.

Example:

- Currency
- Percentage
- Thousands (K)
- Millions (M)

Unlike calculation groups, it only impacts the selected measure.

# Question 4

### Using Tabular Editor, you need to load table data **without rebuilding hierarchies or recalculating calculated columns and measures**.

- ✅ **Process Data**
- Process Default
- Process Defrag
- Process Recalc

## ✅ Correct Answer

**Process Data**

### Explanation

Process modes:

| Mode | Purpose |
|--------|----------|
| **Process Data** | Loads table data only |
| Process Recalc | Recalculates calculated columns/measures |
| Process Default | Runs the required processing automatically |
| Process Defrag | Cleans dictionary storage |

Process Data is the fastest when only loading data.

# Question 6

```DAX
RANKX(
    ALL(Product[Product Name]),
    [Sales],
    ,
    DESC,
    Skip
)
```

What does it calculate?

- Ranks largest sales; ties receive the same rank.
- ✅ **Ranks products by Sales in descending order. Tied products receive the same rank, and the next rank skips numbers according to the number of ties.**
- Smallest sales first with same rank.
- Smallest sales first with unique ranks.

## ✅ Correct Answer

**Ranks the product names by Sales, with the largest values getting the smallest (1,2,3) ranks, and when product names have tied values, the next rank after the tie skips numbers.**

### Explanation

Parameters:

- `DESC` → Highest sales ranked first.
- `Skip` → Ranking skips numbers after ties.

Example:

| Sales | Rank |
|--------|------|
| 100 | 1 |
| 90 | 2 |
| 90 | 2 |
| 80 | 4 |

# Question 8

### Which filter provides the best DAX performance?

- ✅
```DAX
CALCULATE(
    [Sales],
    Calendar[Year] = 2023
)
```

- Filter Sales table
- Filter YEAR(Sales[Date])
- Sales[Year]=2023

## ✅ Correct Answer

```DAX
CALCULATE(
    [Sales],
    Calendar[Year] = 2023
)
```

### Explanation

Filtering the **dimension table** is the most efficient.

Advantages:

- Uses relationship propagation.
- Avoids scanning the large Sales fact table.
- Better storage engine optimization.

Worst option:

```DAX
YEAR(Sales[Date])
```

Applying functions inside filters prevents efficient query optimization.

# Question 9

### Refreshing a table takes over three hours. Only new and changed rows should be refreshed.

- Monthly partitions
- Large semantic model storage
- DirectQuery
- ✅ **Configure Incremental Refresh in Power Query**

## ✅ Correct Answer

**Configure Incremental Refresh in Power Query**

### Explanation

Incremental Refresh:

- Refreshes only recent partitions.
- Leaves historical partitions untouched.
- Greatly reduces refresh duration.

Typical configuration:

- Archive: 3 years
- Refresh: Last 7 days

# Question 10

### Delta table contains 1,000 small Parquet files. Query performance is slow. Keep history for seven days.

- Delete files manually.
- OPTIMIZE only.
- ✅ **Run OPTIMIZE and VACUUM (7-day retention).**
- VACUUM only.

## ✅ Correct Answer

**Select Maintenance and run the OPTIMIZE command as well as the VACUUM command with a retention policy of seven days.**

### Explanation

**OPTIMIZE**

- Compacts many small Parquet files into fewer larger files.
- Improves read performance.

**VACUUM**

- Removes obsolete files no longer needed.
- Frees storage space.
- Retains change history based on the specified retention period (7 days).

Using both commands provides the best balance of performance and storage efficiency.

# 📚 Key Takeaways

| Concept | Remember |
|----------|----------|
| **TREATAS()** | Creates a virtual relationship between unrelated tables |
| **Dynamic Measure Formatting** | Applies DAX-based formatting to a single measure |
| **Large Semantic Model Storage Format** | Required for very large semantic models (e.g., ~50 GB) |
| **Process Data** | Loads table data without recalculating measures or rebuilding hierarchies |
| **Type 2 SCD** | Preserves history by inserting a new row for each change |
| **RANKX (DESC, Skip)** | Highest values rank first; ties share a rank and subsequent ranks are skipped |
| **Performance Analyzer** | Identifies report visuals with the longest rendering time |
| **Filter Dimension Tables** | More efficient than filtering large fact tables in DAX |
| **Incremental Refresh** | Refreshes only new and changed data partitions |
| **OPTIMIZE + VACUUM** | Compacts Delta files and removes obsolete files to improve performance and reduce storage |
